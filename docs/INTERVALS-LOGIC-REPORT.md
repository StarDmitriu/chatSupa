# Логика интервалов в рассылках — полный разбор

## Краткий ответ на вопрос «С какими паузами идёт рассылка?»

Рассылка использует **несколько разных типов пауз**:

| Тип | Где задаётся | Что это | Пример |
|-----|--------------|--------|--------|
| **Окно времени (timeFrom–timeTo)** | Страница «Рассылки» | В какое время суток разрешено ставить отправки (и отправлять) | 00:00 — 23:59 = круглосуточно |
| **Пауза между группами (betweenGroupsSec)** | Страница «Рассылки» → «Паузы…» | **Лимит потока**: секунды между отправками **в разные группы** (rate limit) | TG: 45–90 сек, WA: 45–120 сек |
| **Пауза между шаблонами (betweenTemplatesMin)** | Жёстко в коде при запуске | Минуты между «волнами» шаблонов при **общем** курсе рассылки | 5–15 мин |
| **Интервал/время по группе (send_time)** | У каждой группы в «Группы WA/TG» и в редакторе шаблона | **Когда** в эту группу слать: интервал повтора (2–5 мин, 4 ч, 24 ч) или фиксированное время (HH:mm) | «2–4 часа», «09:00» |
| **Повтор волны (repeat)** | Включён по умолчанию при старте | Через сколько минут создать **новую волну** тех же шаблонов по тем же группам | 5–15 мин |

Ниже — где что хранится, как считается `scheduled_at` и что доработать.

---

## 1. Окно времени (timeFrom, timeTo)

- **Где:** Страница «Рассылки» — кнопка «Время: 00:00 — 23:59», сохраняется в `localStorage` и передаётся в `POST /campaigns/start-multi` как `timeFrom`, `timeTo`.
- **На бэкенде:** сохраняется в `campaigns.time_from`, `campaigns.time_to`; при создании jobs время каждой отправки «зажимается» в это окно (`clampToWindow`). Если окно, например, 08:00–17:00, то все `scheduled_at` попадают только в этот диапазон (в таймзоне пользователя).
- **Итог:** это не «интервал между сообщениями», а **разрешённые часы отправки**.

---

## 2. Пауза между группами (betweenGroupsSecMin / betweenGroupsSecMax)

- **Где:** Страница «Рассылки» → «Паузы…» (TG/WA отдельно). Значения уходят в `POST /campaigns/start-multi`, сохраняются в `campaigns.between_groups_sec_min/max`.
- **Зачем:** Ограничение частоты отправки в разные группы (анти-спам, лимиты API) и отсутствие «залпов».
- **Где используется:** В `createWaveAndEnqueue` как **глобальный лимит потока** — после каждого job общий курсор `cursor` сдвигается на `randInt(betweenGroupsSecMin, betweenGroupsSecMax)` секунд (с учётом `speedFactor` шаблона), независимо от `send_time`.
- **Итог:** это **минимальная пауза между любыми двумя отправками в разные группы**.

---

## 3. Пауза между шаблонами (betweenTemplatesMinMin / betweenTemplatesMinMax)

- **Где:** Страница «Рассылки» → вкладка **Планирование** → блок «Паузы» (ползунок «Пауза между шаблонами»). Значения уходят в `start-multi` и сохраняются в `campaigns.between_templates_min_min/max`.
- **Зачем:** После того как по всем целевым группам «прошла» одна шаблонная волна, перед следующей шаблонной волной добавляется пауза 5–15 минут.
- **Где используется:** В `createWaveAndEnqueue`: между шаблонами (цикл `for (ti...)`) к общему курсору добавляется `randInt(betweenTemplatesMinMin, betweenTemplatesMinMax)` минут. Также для групп с **фиксированным** временем (`send_time = HH:mm`) следующее доступное время в группе сдвигается на этот же интервал (минуты между шаблонами).
- **Итог:** это **интервал между «волнами» по шаблонам** (и сдвиг для фиксированного времени по группе).

---

## 4. Интервал/время по группе (send_time)

- **Что это:** У каждой группы (WA и TG) в БД хранится поле `send_time` в таблицах `whatsapp_groups` и `telegram_groups`. Оно задаёт **как именно** в эту группу распределять отправки.
- **Два варианта значения (бэкенд `parseGroupScheduleSpec`):**
  1. **Фиксированное время** — строка вида `HH:mm` (например `09:00`). Тогда в эту группу сообщения планируются на это время (в пределах окна timeFrom–timeTo), с учётом паузы между шаблонами.
  2. **Интервал** — ключ из `GROUP_INTERVALS`: `2-5m`, `5-15m`, …, `4h`, `6h`, `24h` и т.д. Тогда **интервал — это интервал повтора в эту группу**: следующее сообщение в эту же группу = предыдущее + случайное количество минут из [minMinutes, maxMinutes].
- **Где задаётся в UI:**
  - **Редактор шаблона** (создание/редактирование): для каждой выбранной группы есть выпадающий список «Интервал» (`SEND_INTERVAL_OPTIONS`). При сохранении вызывается `POST /whatsapp/groups/time` или `POST /telegram/groups/time` с `send_time` — обновляется **одно** поле группы в БД.
  - **Управление группами TG:** в таблице есть колонка с `Select` и `SEND_INTERVAL_OPTIONS`; при смене — тот же API `send_time`.
  - **Управление группами WA:** логика `setSendTime` и API есть, но **в таблице нет колонки «Интервал»** — пользователь не может задать интервал для WA-групп в разделе «Группы WA».
- **Поведение:** Один `send_time` на группу для всех шаблонов и всех рассылок. Если в двух шаблонах для одной группы поменять интервал — сохранится последнее сохранённое значение (см. `STYLE-LIQUID-GLASS.md`).
- **Итог:** **Интервал в send_time — это именно интервал повтора отправок в эту группу** (или фиксированное время суток). Не «пауза между группами», а «как часто/когда слать в эту группу».

---

## 5. Повтор волны (repeat)

- **Где:** При запуске рассылки в `basePayload`: `repeatEnabled: true`, `repeatMinMin: 5`, `repeatMinMax: 15`. Сохраняется в `campaigns.repeat_enabled`, `repeat_min_min`, `repeat_min_max`, `next_repeat_at`.
- **Как работает:** Сразу после старта считается `next_repeat_at = now + rand(5..15) мин`. Сервис `CampaignRepeatService` раз в 10 сек смотрит кампании с `repeat_enabled=true` и `next_repeat_at <= now`; для них вызывается `repeatWaveIfReady` — создаётся **новая волна** jobs (те же шаблоны, те же группы) и ставится следующий `next_repeat_at`.
- **Итог:** Это **интервал повтора всей «волны»** (все шаблоны по всем группам снова), не интервал между сообщениями внутри одной волны.

---

## Сводная схема: как считается scheduled_at для одного job

1. Берутся группы (с `send_time`) и шаблоны, строится список пар (шаблон, группа).
2. Для каждой пары:
   - **Если у группы send_time = интервал (например 2–4h):** для этой группы ведётся «следующее доступное время»: либо от последней запланированной отправки в эту группу + случай из интервала, либо от base. `scheduled_at` = это время, зажатое в окно timeFrom–timeTo.
   - **Если send_time = HH:mm:** следующая отправка в группу в это время (в окне); следующее доступное время сдвигается на betweenTemplatesMin.
   - **В любом случае:** итоговое `scheduled_at` = максимум из «можно по группе» и «можно по потоку (cursor)»; после каждого job курсор сдвигается на betweenGroupsSec (секунды).
3. Между шаблонами курсор сдвигается на betweenTemplatesMin (минуты).

Таким образом, **рассылка идёт по всем выбранным группам и шаблонам**, а паузы такие:
- между **разными группами** (лимит потока, всегда) — 45–90/120 сек (или то, что выставлено в «Паузы…»);
- между **шаблонами** — 5–15 мин;
- внутри **одной группы** при интервале send_time — по интервалу (2–5 мин, 4 ч, 24 ч и т.д.);
- **повтор всей волны** — каждые 5–15 мин (если repeat включён).

## Что доработать и улучшить

### 1. Показать и дать править паузы на странице «Рассылки»

- **Сейчас:** `betweenGroupsSec` и `betweenTemplatesMin` зашиты в коде; окно времени — единственное, что пользователь видит и меняет.
- **Имеет смысл:** Вынести в UI (хотя бы продвинутый блок «Паузы»):
  - пауза между группами (сек): мин–макс (с подсказкой: «между отправками в разные группы»);
  - пауза между шаблонами (мин): мин–макс;
  - включение/выключение повтора волны и интервал повтора (мин).
- Так пользователь будет понимать, «с какими пазами» идёт рассылка, и сможет ослабить/усилить нагрузку.

### 2. Добавить колонку «Интервал» в разделе «Группы WA»

- **Сейчас:** В «Группы TG» есть выбор интервала (`SEND_INTERVAL_OPTIONS`); в «Группы WA» — только выбор «выбрано/не выбрано», `setSendTime` и API есть, но колонки с Select нет.
- **Нужно:** Добавить колонку «Интервал» в таблицу групп WA с тем же `Select` и `SEND_INTERVAL_OPTIONS`, по аналогии с TG, чтобы интервал по группе можно было задать в одном месте.

### 3. Поддержка фиксированного времени (HH:mm) во фронте

- **Сейчас:** Бэкенд уже поддерживает `send_time = "09:00"`; во фронте в `SEND_INTERVAL_OPTIONS` только интервалы (2–5m, 4h, …).
- **Нужно:** В редакторе шаблона и в управлении группами добавить возможность задать «Фиксированное время» (TimePicker или выбор HH:mm), чтобы записывать в `send_time` строку вида `HH:mm`.

### 4. Один источник правды для опций интервала

- **Сейчас:** Список интервалов: бэкенд `GROUP_INTERVALS` в `campaigns.service.ts`, фронт `SEND_INTERVAL_OPTIONS` в `constants/sendIntervals.ts`. Нужно следить, чтобы ключи совпадали (например уже есть `4h` в обоих).
- **Имеет смысл:** Либо опции с бэкенда отдавать в API (например в настройках/профиле), либо в одном месте на фронте держать константу и при добавлении нового варианта (например «раз в 8 часов») добавлять и в бэкенд, и во фронт — и описать это в комментарии/документации.

### 5. Подсказки в UI

- На странице «Рассылки» рядом с окном времени добавить короткую подсказку: «Окно отправки по времени суток. Паузы между группами и шаблонами заданы по умолчанию (можно вынести в настройки).»
- В редакторе шаблона у поля «Интервал» группы: «Интервал повтора отправок в эту группу (или фиксированное время). Задаётся один раз на группу для всех шаблонов.»

### 6. Документация для пользователя

- В разделе «Помощь» или в подсказке на странице «Рассылки» кратко описать:
  - что такое окно времени;
  - что такое интервал по группе (повтор в эту группу);
  - что рассылка идёт по всем выбранным группам и шаблонам с паузами между группами и между шаблонами (и при желании — где эти паузы посмотреть/настроить после доработки п.1).
