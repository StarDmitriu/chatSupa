# Рассылки: отдельный воркер, вынос WA, приоритеты и лимиты

Кратко **зачем** это вообще обсуждается и **когда** имеет смысл тратить на это инженерное время.

---

## 1. Отдельный процесс только под воркер (BullMQ)

**Идея:** второй процесс Node (или второй процесс PM2), в котором поднимается **только** обработчик очереди `campaign-send-*`, без полного HTTP API Nest.

**Зачем**

- **Изоляция нагрузки:** тяжёлая отправка (скачивание медиа, ожидание Telegram 90 с, Baileys) не делит event loop с запросами `/auth/me`, кабинета и дашборда.
- **Изоляция падений:** утечка или необработанное исключение в воркере не роняет API сразу (и наоборот — рестарт API не обязан останавливать обработку очереди, если воркер отдельный).
- **Масштаб по горизонтали:** можно поднять **несколько** воркеров на **разные** префиксы Redis или разные очереди (осторожно с дублированием обработки одной очереди двумя воркерами без координации).

**Когда имеет смысл**

- Десятки/сотни одновременных job’ов, заметные пики CPU/задержки API при активных рассылках.
- Требования к **uptime** API независимо от «хвоста» очереди.

**При ~10–20 онлайн-пользователях** с уже введённым **шардированием очередей** отдельный процесс воркера чаще **не обязателен** — это следующий шаг после метрик (если видно, что API страдает синхронно с очередью).

**Реализация в коде:** отдельный `main.worker.ts` + урезанный Nest-модуль (только `QueueModule`, `WhatsappModule`, `TelegramModule`, `SupabaseModule`, …) и вторая строка в PM2. Сейчас в репозитории **не сделано** намеренно (объём рефакторинга).

## 2. Вынос WhatsApp (Baileys) в отдельный сервис

**Идея:** процесс(ы), которые **только** держат сокеты Baileys и отвечают на внутренний RPC/HTTP «отправь в группу X», а основной backend остаётся «тонким».

**Зачем**

- **Память и стабильность:** Baileys тяжёлый; при 50–100 WA в одном процессе проще перезапускать **часть** воркеров WA, а не весь монолит.
- **Масштаб:** несколько машин под WA, один Redis/очередь для команд.
- **Граница ответственности:** проще ограничивать ресурсы контейнером только под WA.

**Когда имеет смысл**

- Много **постоянно подключённых** WA-аккаунтов в продакшене, рост RAM выше комфортного для одного Node.

**При 10–20 онлайн** обычно **не требуется**. Альтернатива по продукту — **официальный WhatsApp Cloud API** (другая архитектура и договорённости с Meta).

**В репозитории не реализовано** — это отдельный крупный проект.

## 3. Приоритетные очереди и лимит «на пользователя»

**Приоритеты (VIP)**

- **Зачем:** важные клиенты не ждут в хвосте общей очереди шарда, когда перед ними сотни сообщений «обычных» пользователей на том же шарде (коллизия хэша) или когда в одной очереди смешаны разные пользователи.
- **Реализация:** в BullMQ у job есть поле **`priority`** (0 — наивысший приоритет). Можно помечать job’ы VIP-пользователей меньшим числом.
- **Ограничение:** приоритет влияет на порядок **внутри одной очереди** BullMQ; шардирование по `userId` у вас уже разводит большинство пользователей по разным очередям.

**Лимит / пауза по пользователю**

- У вас уже есть: **не более одной активной отправки на `userId`** (`runSerializedByUser`), ритм по кампании (`enforceCampaignSendRhythm`), отдельные ретраи FLOOD/коннекта для TG/WA.
- Дополнительный **глобальный минимальный интервал между любыми отправками одного пользователя** имеет смысл, если нужно **жёстче** соблюдать анти-спам политику платформы — это отдельная политика продукта.

**Что сделано в коде:** `CampaignVipService` объединяет **`CAMPAIGN_VIP_USER_IDS`** и колонку **`users.campaign_send_vip`** (миграция Supabase). Периодическое обновление из БД — `CAMPAIGN_VIP_REFRESH_INTERVAL_MS`. Подробно: [CAMPAIGN_QUEUE_RUNBOOK.md](./CAMPAIGN_QUEUE_RUNBOOK.md).

## 4. Итог для вашего масштаба

| Мера                         | ~10–20 онлайн      |
|-----------------------------|--------------------|
| Шардирование очередей       | Уже есть           |
| Отдельный процесс воркера   | Обычно не нужен    |
| Отдельный сервис WA         | Обычно не нужен    |
| VIP-приоритет в очереди     | По желанию (env)   |

Дальнейший шаг при росте — **метрики** (`/health/campaign-queues` + логи), затем решение по отдельному воркеру **по фактам** нагрузки.
