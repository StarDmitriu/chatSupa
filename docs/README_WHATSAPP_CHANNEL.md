# WhatsApp-канал в проекте: термины, архитектура, перенос, развитие

Документ описывает, **как в этом репозитории** устроено подключение WhatsApp, какие термины и библиотеки используются, что переносить на новый сервер и куда смотреть при эволюции в полноценный клиент.

---

## 1. Глоссарий терминов

| Термин | Значение |
|--------|----------|
| **RTT** (round-trip time) | Время кругового пакета/запроса «туда–обратко». В UI для зонда — сколько миллисекунд занял лёгкий HTTP `HEAD` до `https://web.whatsapp.com/`. |
| **HTTP HEAD** | Запрос как GET, но без тела страницы — только заголовки. Нужен для быстрой проверки «достучались ли до хоста». |
| **WebSocket** | Долгое двунаправленное соединение. Протокол чата WhatsApp с серверами Meta идёт по своему каналу, не как обычная загрузка веб-страницы. |
| **Зонд / probe («До web.whatsapp.com»)** | Отдельная проверка на бэкенде: один `HEAD` на публичный веб-хост WhatsApp. **Не равна** состоянию сокета Baileys к серверам чата. |
| **408 (Baileys / disconnect)** | Код закрытия соединения в обвязке библиотеки; часть «transient» сценариев обрабатывается как временный обрыв с авто-ретраями. Не путать с HTTP 408 вашего REST API. |
| **ETIMEDOUT** | Сетевая ошибка: операция не уложилась в таймаут (подключение, чтение, запись). |
| **Backoff / jitter** | Пауза перед следующей попыткой; **jitter** — случайное смещение, чтобы не синхронно нагружать сервер. |
| **Creds / `creds.json` / папка сессии** | Файлы авторизации Baileys под пользователя (в проекте: каталог `wa_auth/<userId>/`). Пока сессия валидна у WhatsApp, можно переподключаться без нового QR. |
| **QR-привязка** | Одноразовое связывание телефона с «связанным устройством» через сканирование QR в приложении WhatsApp. |
| **JID** | Идентификатор в сети WhatsApp, например `79991234567@s.whatsapp.net` или группа `…@g.us`. |
| **wa_id** | Идентификатор/номер аккаунта в удобном для отображения виде (часть из JID). |
| **Baileys** | npm-пакет `@whiskeysockets/baileys` — реализация клиента протокола WhatsApp Web на Node.js. **Не** официальный WhatsApp Business Platform (Cloud API). |
| **NestJS** | Фреймворк backend: модули, контроллеры, DI. Основная логика: `backend/src/whatsapp/whatsapp.service.ts`. |
| **Next.js** | Frontend на React. Блок кабинета: `frontend/src/components/WhatsappConnectBlock.tsx`. |
| **PM2** | Менеджер процессов Node: `pm2 restart backend` / `frontend`. |

---

## 2. Как связано в коде

1. **Backend** (`WhatsappService`): для каждого `userId` — сессия Baileys, сокет, статусы (`connected`, `connecting`, `pending_qr`, `temporary_network_issue`, `error`, `not_connected`), ретраи при transient 408, зонд до `web.whatsapp.com`, папка `wa_auth/<userId>/`.
2. **HTTP API** (контроллер `whatsapp`): `/whatsapp/start`, `/whatsapp/status/:userId`, `/whatsapp/disconnect`, `account-info`, `account-avatar`, и т.д.
3. **Frontend**: опрос статуса, QR, текст восстановления, диагностика RTT зонда, карточка аккаунта, отключение.
4. **Рассылки**: кампании и воркер BullMQ проверяют подключение и отправляют сообщения через тот же сервис; очередь заданий в БД/Redis, не «внутри браузера».

Ключевые файлы:

- `backend/src/whatsapp/whatsapp.service.ts`
- `backend/src/whatsapp/whatsapp.controller.ts`
- `frontend/src/components/WhatsappConnectBlock.tsx`

## 3. Перенос на другой сервер (чеклист)

1. **Код** — git clone / копия репозитория, те же ветки.
2. **Секреты** — `.env` на новом хосте (не коммитить): JWT, Supabase, Redis, ключи WA при необходимости, и т.д.
3. **Данные сессий WhatsApp** — каталог **`wa_auth/`** у backend (или путь из кода: `process.cwd()/wa_auth/<userId>`). Без него пользователям придётся сканировать QR заново.
4. **Инфраструктура** — Node.js, Redis (очереди), Supabase/Postgres, сборка `npm run build` в `backend` и `frontend`.
5. **Сеть** — исходящий HTTPS/WebSocket до инфраструктуры WhatsApp; новый IP может вести себя иначе (фильтры, маршрут).
6. **Запуск** — например `pm2` с `node dist/main` для backend и standalone Next для frontend (как у вас сейчас).

## 4. Ограничения и риски

- Протокол через **Baileys** — это не контракт Meta **Cloud API**; для крупного B2B часто рассматривают официальный API отдельно.
- **Полностью убрать** обрывы по таймауту между вашим сервером и сетью WhatsApp **нельзя гарантировать**: цепочка «ваш VPS → интернет → Meta» вне полного контроля. Можно **снижать** частоту и время восстановления (сеть, таймауты, ретраи, мониторинг).

## 5. Перспектива: к полноценному «мессенджер-клиенту» + ИИ

| Направление | Идея |
|-------------|------|
| **Надёжность** | Метрики времени до `open`, счётчики 408, алерты; при необходимости дополнительные проверки (осторожно с частотой). |
| **Продукт «чаты»** | Списки диалогов, история, вложения, поиск — отдельная модель + подписка на события Baileys (`messages.upsert` и др.). |
| **ИИ** | Входящее → очередь → модель → исходящее; идемпотентность, лимиты WhatsApp, модерация. |
| **Масштаб** | Много пользователей → выделенные воркеры, привязка сессии к процессу, бэкапы `wa_auth`. |
