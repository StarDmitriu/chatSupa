# Отчёт: частотности, лимиты и логики рассылок TG/WA

## 1. Текущая архитектура

### 1.1 Очередь (BullMQ)
- **Очередь:** `campaign-send`
- **Concurrency:** 1 (одна задача одновременно)
- **Порядок:** Jobs добавляются с `delay = scheduled_at - now`, обрабатываются по времени готовности → **порядок соблюдается**
- **Retry:** `attempts: 5`, `backoff: { type: 'exponential', delay: 5000 }` — при ошибке до 5 повторов с задержкой 5s, 10s, 20s, 40s, 80s

### 1.2 Планирование волн
- **createWaveAndEnqueue** создаёт jobs с `scheduled_at` на основе:
  - `between_groups_sec_min/max` — пауза между группами (сек)
  - `between_templates_min_min/max` — пауза между шаблонами (мин)
  - `send_time` группы — фиксированное время (HH:MM) или интервал (2-5m, 5-15m и т.д.)

### 1.3 Повтор волн (repeat)
- **CampaignRepeatService** каждые 10 сек проверяет кампании с `repeat_enabled=true` и `next_repeat_at <= now`
- При срабатывании создаётся новая волна с теми же группами/шаблонами
- Интервал повтора: `repeat_min_min`–`repeat_min_max` минут (по умолчанию 5–15 из UI)

## 2. Частотности в UI и бэкенде

### 2.1 Страница кампаний (`/dashboard/campaigns`)
При старте рассылки передаются:

| Параметр | Telegram | WhatsApp |
|----------|----------|----------|
| betweenGroupsSecMin | 30 | **126** |
| betweenGroupsSecMax | 60 | **400** |
| betweenTemplatesMinMin | 5 | 5 |
| betweenTemplatesMinMax | 15 | 15 |
| repeatMinMin | 5 | 5 |
| repeatMinMax | 15 | 15 |

**Проблема:** WA использует 126–400 сек (2–7 мин) между группами — очень медленно. TG — 30–60 сек. Telegram обычно строже по flood, но 30–60 сек для TG при большом числе групп может давать FloodWait.

### 2.2 Интервал по группам (send_time)
- **Бэкенд:** `telegram_groups.send_time` и `whatsapp_groups.send_time` поддерживаются
- **UI в шаблоне:** селектор интервала показывается **только для TG** (`channel === 'tg'`)
- **Проблема:** для WA групп нет выбора интервала в редакторе шаблона — все WA группы идут по общему расписанию (cursor + betweenGroupsSec)

### 2.3 Дефолты в campaigns.service
Если фронт не передаёт задержки:
- `between_groups_sec_min = 2`, `between_groups_sec_max = 3` — **2–3 сек между группами**
- Страница кампаний всегда передаёт свои значения, поэтому эти дефолты не используются при старте. Но при requeue или repeat используются значения из БД кампании.

## 3. Отправка: TG vs WA

### 3.1 Telegram (sendToGroup)
- Клиент: GramJS (TelegramClient)
- **FloodWait:** GramJS обычно обрабатывает FloodWait внутри (ждёт и повторяет)
- **Таймаут worker:** 30 сек — при долгом FloodWait (>30 сек) возможен timeout
- При ошибке — throw → BullMQ делает retry

### 3.2 WhatsApp (sendToGroup)
- Клиент: Baileys (whatsapp-web.js)
- Нет встроенной обработки rate limit
- При ошибке — throw → BullMQ делает retry
- **Одинаковая логика retry** для TG и WA на уровне BullMQ

### 3.3 Повтор отправки
- **TG:** есть retry на уровне BullMQ + возможная внутренняя обработка FloodWait в GramJS
- **WA:** retry только через BullMQ
- Оба канала используют `attempts: 5` и exponential backoff

## 4. Рекомендуемые лимиты (по публичным данным)

### Telegram (user accounts, MTProto)
- ~20 сообщений/мин в разные чаты
- ~1 сообщение / 3 сек в одну группу
- Рекомендация: **минимум 3–5 сек между группами**, лучше 15–30 сек для безопасности

### WhatsApp (user accounts, whatsapp-web.js)
- Нет официальных лимитов для user API
- Практика: 1–2 сообщения/сек, но лучше **30–60 сек** между группами для снижения риска бана

## 5. Выявленные проблемы и рекомендации

### 5.1 Критичные
1. **Интервал только для TG в UI шаблона**  
   - Для WA групп нет селектора `send_time` в редакторе шаблона  
   - **Рекомендация:** показывать интервал и для WA (`channel === 'wa'`)

2. **Слишком быстрые дефолты в бэкенде**  
   - `between_groups_sec_min=2`, `max=3` — опасно при прямом вызове API  
   - **Рекомендация:** поднять до 15–30 сек

### 5.2 Рекомендуемые изменения частот
1. **Telegram:** 30–60 сек — приемлемо, но при 50+ группах лучше 45–90 сек
2. **WhatsApp:** 126–400 сек — избыточно медленно; можно 45–120 сек

### 5.3 Retry и таймауты
1. **Таймаут 30 сек** — при FloodWait >30 сек TG будет падать по timeout  
   - Рекомендация: увеличить до 90–120 сек для TG или обрабатывать FloodWait явно
2. **Retry:** логика одинаковая для TG и WA, BullMQ retry работает для обоих

### 5.4 Повтор волн (repeat)
- Интервал 5–15 мин — разумно
- `CAMPAIGN_REPEAT_TICK_MS=10000` — проверка каждые 10 сек, ок

### 5.5 Очередность
- Jobs добавляются с `delay` по `scheduled_at`
- Worker concurrency=1 → порядок соблюдается
- При repeat новая волна создаёт новые jobs с `scheduled_at` в будущем — порядок сохраняется

## 7. Масштабирование: 10 пользователей одновременно

### 7.1 Архитектура при 10 пользователях

| Компонент | Текущее состояние | При 10 пользователях |
|-----------|------------------|----------------------|
| **Worker** | 1 процесс, concurrency: 1 | Один worker обрабатывает все jobs |
| **Очередь** | Одна `campaign-send` | Все 10 пользователей в одной очереди |
| **Порядок** | По `scheduled_at` (delay) | Сохраняется — jobs обрабатываются по времени готовности |
| **TG/WA сессии** | Map userId → client | 10 TG + до 10 WA сессий в памяти (~500 MB) |

### 7.2 Пропускная способность

- **1 job за раз** → ~1 сообщение каждые 45–90 сек (с учётом задержек)
- **10 пользователей × 50 групп × 3 шаблона** = 1500 jobs на волну
- **Время волны:** 1500 × 60 сек ≈ **25 часов** (при 45–90 сек между группами)
- **Repeat:** каждые 5–15 мин добавляется новая волна → очередь растёт быстрее, чем обрабатывается

**Вывод:** При 10 активных пользователях очередь будет накапливаться. Рассылки выполняются, но с большой задержкой.

### 7.3 Потенциальные проблемы

1. **lockDuration (критично)**  
   - Дефолт BullMQ: 30 сек. Таймаут отправки TG: 90 сек.  
   - Job может быть помечен как stalled до завершения → лишние retry.  
   - **Решение:** `lockDuration: 120_000` в Worker.

2. **CampaignRepeat tick overlap**  
   - Tick каждые 10 сек. Обработка 10 кампаний может занять 20+ сек.  
   - Следующий tick стартует до завершения предыдущего.  
   - **Риск:** Дополнительная нагрузка на БД. Claim в `repeatWaveIfReady` защищает от дублей.

3. **WA startSession**  
   - Если WA не подключён, `startSession` возвращается быстро (не ждёт QR).  
   - Job падает с `wa_not_connected`, BullMQ делает retry.  
   - Очередь не блокируется.

4. **Redis**  
   - Одна очередь, десятки тысяч jobs — Redis справляется.  
   - Один экземпляр Redis — единая точка отказа.

5. **PM2 instances: 1**  
   - Один процесс бэкенда → один worker. Дубликатов нет.

### 7.4 Рекомендации для 10+ пользователей

1. Увеличить **concurrency** до 2–3 (разные пользователи = разные аккаунты, rate limit не общий).
2. Добавить **lockDuration: 120_000** в Worker.
3. При росте нагрузки — несколько инстансов backend с общей Redis (BullMQ распределит jobs).
4. Мониторинг длины очереди и времени обработки.

### 7.5 Итог: будет ли работать при 10 пользователях?

| Аспект | Статус |
|--------|--------|
| Ошибки/падения | Нет — архитектура выдерживает |
| Дубликаты сообщений | Нет — status check + lock защищают |
| Порядок отправки | Соблюдается по `scheduled_at` |
| Блокировка очереди | Нет — WA startSession не блокирует |
| Пропускная способность | Низкая — ~1 msg/мин на всех |
| Рост очереди | Да — при repeat быстрее, чем обработка |

**Вывод:** При 10 пользователях рассылки будут выполняться корректно, без ошибок и дублей, но очередь будет расти. Для ускорения — увеличить concurrency или добавить инстансы.
