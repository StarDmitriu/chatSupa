# Отладка ошибок оплаты (Prodamus)

## Где взять доступы (FORM_URL, SECRET_KEY, SYS)

Всё берётся в **личном кабинете мерчанта Prodamus** (платёжная страница, куда вы заходите как владелец):

1. **Войти** на платёжную страницу Prodamus (тот же домен, что вы используете для приёма оплаты).
2. Внизу открыть **«Настройки»**.
3. **Секретный ключ** — в блоке «Секретный ключ» скопировать значение → это `PRODAMUS_SECRET_KEY`.
4. **URL формы** — это адрес вашей платёжной страницы с завершающим слэшем, например `https://payform.prodamus.ru/` или ваш кастомный домен → `PRODAMUS_FORM_URL`.
5. **Идентификатор системы (sys)** — в настройках или в URL/документации вашего договора с Prodamus → `PRODAMUS_SYS`.

Если личного кабинета ещё нет или нужен новый ключ — обращение в поддержку Prodamus: 8 (495) 150-08-71, sales@prodamus.ru, Telegram @prodamus_bot. Документация: https://help.prodamus.ru/payform/integracii/rest-api

## Что проверять при ошибке при оплате

### 1. Переменные окружения бэкенда

В `backend/.env` должны быть заданы (без них создание ссылки падает):

- `PRODAMUS_FORM_URL` — базовый URL формы (например `https://payform.prodamus.ru/`)
- `PRODAMUS_SECRET_KEY` — секретный ключ для подписи
- `PRODAMUS_SYS` — идентификатор системы в Prodamus

Проверка: при старте бэкенда не должно быть `Missing env: PRODAMUS_*`.

### 2. Логи бэкенда

При ошибке создания платежа в логах появятся строки:

- `[Prodamus create] users error: ...` — ошибка загрузки пользователя из БД
- `[Prodamus create] payments insert error: ...` — ошибка вставки в таблицу `payments`
- `[Prodamus create] Prodamus link failed: ...` — Prodamus вернул не 200 или не вернул ссылку
- `[Prodamus create] Prodamus не вернул ссылку на оплату. Ответ: ...` — в ответе нет URL

Просмотр: `pm2 logs backend --err --lines 100` или `tail -f ~/.pm2/logs/backend-error-0.log`.

### 3. Ответ Prodamus (do=link)

Бэкенд делает GET-запрос к форме Prodamus с параметрами заказа. Ожидается:

- статус 200;
- в теле — либо сразу URL (строка), либо JSON с полем `link`, `url`, `payment_url` или `redirect_url`.

Если Prodamus возвращает HTML-страницу с ошибкой или другой формат — бэкенд вернёт `payment_provider_error`, а в логах будет фрагмент ответа.

### 4. Фронтенд

Пользователь видит сообщения по коду ответа API:

| Код ответа API                    | Сообщение пользователю |
|-----------------------------------|-------------------------|
| `user_not_found`                  | Пользователь не найден. Войдите снова. |
| `invalid_plan_code`               | Выбран неверный тариф. |
| `supabase_users_error`           | Ошибка загрузки данных. Попробуйте позже. |
| `supabase_payments_insert_error` | Не удалось создать платёж. Попробуйте позже или обратитесь в поддержку. |
| `payment_provider_error`         | Ошибка платёжной системы. Попробуйте позже или оплатите другим способом. |

В ответе API может быть поле `error` с деталями — оно подставляется в уведомление, если для кода нет готовой фразы.

### 5. Таблица payments (Supabase)

Проверьте, что при нажатии «Оплатить» создаётся запись в `payments` со статусом `created` и заполненным `order_id`. Если записи нет — смотрите `payments insert error` в логах и RLS/права в Supabase.

### 6. Webhook после оплаты

После успешной оплаты Prodamus шлёт webhook на `POST /api/payments/prodamus/webhook`. В логах бэкенда должна быть строка вида:

`[Prodamus webhook] order_num=... order_id=... payment_status=...`

Если webhook не доходит — проверьте URL в настройках Prodamus и доступность бэкенда с интернета.
