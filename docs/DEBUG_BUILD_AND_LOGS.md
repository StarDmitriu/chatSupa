# Проверка билда и логирование

## Зачем

Чтобы убедиться, что на всех страницах отдаётся **один и тот же билд** (нет старого кэша и не крутятся несколько инстансов с разными билдами).

## Что добавлено

1. **Идентификатор билда** — при каждой сборке (`npm run build`) в приложение подставляется `NEXT_PUBLIC_BUILD_ID` (например `build-20250307142305` или значение из env).
2. **В браузере:**
   - В консоли (F12 → Console) при загрузке любой страницы выводится: `[ЧатРассылка] build=... path=...`
   - На элементе `<html>` есть атрибут `data-build-id` — можно проверить в DevTools (Elements).
3. **В ответах сервера:** заголовок **`X-Build-Id`** — по нему можно проверить билд без браузера (curl, DevTools → Network).
4. **Логи сервера:** для каждого запроса к странице (не к `/_next/static`, не к `/api`) в stdout пишется строка:
   ```
   [2025-03-07T12:00:00.000Z] GET / host=chatrassylka.ru build=build-20250307142305
   ```

## Как проверить «один билд на всех страницах»

### Вариант 1: curl по заголовку

После деплоя выполните (подставьте свой домен):

```bash
echo "Главная:"
curl -sI https://chatrassylka.ru/ | grep -i x-build-id

echo "Кабинет:"
curl -sI https://chatrassylka.ru/cabinet | grep -i x-build-id

echo "Дашборд группы:"
curl -sI "https://chatrassylka.ru/dashboard/groups" | grep -i x-build-id
```

Значение `X-Build-Id` на всех ответах должно быть **одинаковым**. Если на одной странице один билд, на другой — другой, возможны несколько инстансов или старый кэш (CDN/прокси).

### Вариант 2: браузер

1. Откройте https://chatrassylka.ru/ → F12 → Console. Должно быть: `[ЧатРассылка] build=... path=/`
2. Перейдите на https://chatrassylka.ru/cabinet — снова посмотрите в консоль: `path=/cabinet`, **build=** должен быть тот же.
3. Аналогично проверьте `/dashboard/groups`, `/dashboard/telegram-groups` и т.д.

Если на разных страницах разный `build` — значит запросы уходят на разные инстансы или отдаётся закэшированный старый HTML.

### Вариант 3: логи процесса (PM2 / Docker)

При запросах к страницам в логах фронтенда должны появляться строки с путём и билдом, например:

```
[2025-03-07T12:00:00.000Z] GET / host=chatrassylka.ru build=build-20250307142305
[2025-03-07T12:00:01.000Z] GET /cabinet host=chatrassylka.ru build=build-20250307142305
```

- **PM2:** `pm2 logs frontend`
- **Docker:** `docker logs -f chatrassylka_frontend`

По ним видно, какой билд отдаёт текущий процесс и какие пути запрашивают.

## Несколько билдов / инстансов

Если видите разный `X-Build-Id` на разных запросах:

1. **Один хост, один порт (3001):** убедитесь, что запущен только один процесс фронтенда (один контейнер или один инстанс PM2). Два процесса на одном порту не уживутся; если порты разные — nginx мог проксировать на старый инстанс.
2. **Кэш:** проверьте, не кэширует ли nginx или CDN ответы по `location /` (для SPA/Next лучше не кэшировать HTML или кэшировать с коротким TTL).
3. **Деплой:** после `npm run build` и перезапуска контейнера/PM2 подхватывается новый билд; пока не перезапустили — отдаётся старый.

## Задать свой BUILD_ID при сборке

Чтобы явно проставить метку (например, из CI):

```bash
NEXT_PUBLIC_BUILD_ID=release-1.2.3 npm run build
```

Тогда в логах и в заголовке будет `release-1.2.3`.
