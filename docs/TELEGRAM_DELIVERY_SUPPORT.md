# Поддержка: ошибки доставки в Telegram (рассылки)

Краткий сценарий для операторов и разработчиков. **Шаблоны рассылки и привязки групп к шаблонам не трогаем как «первый шаг»** — сначала восстанавливаем актуальность данных Telegram и доступ аккаунта к чатам.

---

## Частые ошибки

### `CHANNEL_INVALID` / `PEER_ID_INVALID`

**Смысл:** сервер Telegram не принимает пару «id чата + доступ» (peer), с которой идёт отправка. Часто это устаревший `access_hash` в БД, миграция группы в супергруппу, выход из чата или удалённый чат.

**Что делать пользователю / в ЛК:**

1. **Синхронизация списка Telegram-групп** (кнопка в кабинете) — обновляет строки в `telegram_groups` из живых диалогов.
2. Убедиться, что **аккаунт Telegram по-прежнему состоит** в нужном чате и **может писать** (не только читать).
3. При подозрении на сессию: **переподключить** Telegram (QR/телефон), затем снова **синхронизация**.

**Не делаем по умолчанию:** не просим «удалить чаты из шаблона» — это ломает задуманный охват. Система может **автоматически снять с выбора** проблемный чат после нескольких повторяющихся ошибок (см. воркер кампаний, `TG_AUTO_UNSELECT_*`).

### `CHANNEL_PRIVATE`, `CHAT_WRITE_FORBIDDEN`, `USER_BANNED_IN_CHANNEL`

Нет прав на отправку или нет доступа к чату — проверка членства, ролей, бана, закрытых тем в форумах.

### `FLOOD_WAIT`

Лимит Telegram — рассылка сама откладывает job; пользователю: не уменьшать искусственно паузы до минимума на больших объёмах.

---

## Что сделано в бэкенде (самовосстановление)

- После успешного резолва супергруппы/канала **`tg_access_hash` сохраняется в `telegram_groups`**, чтобы реже опираться на устаревший снимок.
- При **`CHANNEL_INVALID` / `PEER_ID_INVALID`** выполняется **один** цикл: `getDialogs` + пересборка peer + повторная отправка.
- Результат retry теперь логируется отдельными событиями (`tg_peer_refresh_retry_started|success|failed`) в `limit_learning_events`.
- При **синхронизации групп** флаг **`is_selected` не затирается** для чатов, которые уже были в БД: сохраняется предыдущий выбор (новые диалоги по-прежнему попадают с выбором «включено» по умолчанию).
- Для проблемных чатов включён **soft quarantine**: временная блокировка по порогу повторных permanent-fail с авто-реактивацией после успешного sync.

## Runbook TG incident (1-page)

1. **Проверить сигналы**
   - `npm run check:campaign-invariants`
   - `/health/campaign-queues` и `/admin/campaigns/diagnostics`
   - признаки: `tg_peer_errors_spike_1h`, рост `due_pending`, `processing=0`.
2. **Сделать sync + preflight**
   - `POST /telegram/sync-groups`
   - `GET /campaigns/preflight/tg`
   - если `badRate` выше порога, запуск волны откладываем и чиним доступы/peer.
3. **Перезапустить волну в incident-профиле**
   - requeue/restart кампании после sync
   - включить incident-профиль, если детектор переключил режим автоматически.

Важно: **не удалять группы из шаблонов вручную как первую реакцию**.
