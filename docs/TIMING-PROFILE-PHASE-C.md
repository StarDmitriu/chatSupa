# Серверный профиль тайминга (фаза C)

Кратко: **вынести «профиль отправки» на бэкенд**, чтобы настройки были привязаны к **аккаунту**, а не к одному браузеру.

## Зачем

Сейчас часть настроек (окно суток, паузы, повтор волны, фрагменты UI планирования и т.п.) живёт только в **`localStorage` / `sessionStorage`**. На другом устройстве, в другом браузере или после очистки данных сайта они **не подтянутся** — в БД на сервере их нет.

Фаза C даёт:

- **Единые настройки** на всех устройствах;
- **Бэкап** на сервере;
- Меньше саппорта в духе «сбросил кэш — всё пропало».

## Что предполагается (черновик)

- API вроде **`GET/PUT /user/send-timing-profile`** — JSON с дефолтами волны (окно, паузы, повтор и т.д.).
- Клиент при **логине** подгружает сохранённый профиль и кладёт в интерфейс; при **изменении** — сохраняет на сервер.
- **Опционально** — массовое обновление `send_time` по группам (тяжёлая и **рискованная** операция: может затронуть шаблоны/ожидания).
- Альтернатива — **мастер первого запуска**: один сценарий сразу настраивает кампанию и дефолты **с записью на сервер**.

## Почему «ещё не делалось»

Это **отдельная задача на бэкенд**: схема БД, миграции, права доступа, согласование с тем, что уже хранится в **campaigns / группах / шаблонах**. Фронт можно улучшать без этого API; фаза C — следующий этап, если нужна «настоящая одна форма» на уровне аккаунта.
